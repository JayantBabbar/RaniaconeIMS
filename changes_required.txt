## REQ-1: GET /tenants ignores `limit` query parameter, hard-caps at 100

### Severity
HIGH — silently truncates results once the deployment exceeds 100 tenants.

### Observed
- DB has 136 active tenants.
- FE sends `GET /tenants?limit=200`.
- Response is exactly 100 rows, no pagination envelope.

### Root cause (located)
`packages/auth/app/api/v1/tenants.py`:

```python
@tenants_router.get("", response_model=list[TenantResponse], ...)
async def list_tenants(
    db: AsyncConnection = Depends(get_db),         # ← no Query params declared
) -> list[TenantResponse]:
    rows = await tenant_repo.list_all(db)          # ← no limit forwarded
    return [TenantResponse(**r) for r in rows]
```

`packages/auth/app/repositories/tenant_repo.py`:

```python
async def list_all(conn: AsyncConnection, *, limit: int = 100) -> ...:
    "ORDER BY created_at DESC LIMIT %s"
```

The handler doesn't declare `limit` so FastAPI never passes it through.
Repo defaults to 100. Result: cap of 100 regardless of client request.

### Fix asked
Either:
- (Preferred) Make `list_tenants` accept
  `limit: int = Query(100, le=500)` and
  `cursor: UUID | None = Query(None)` and pipe both to the repo, matching
  the pattern other list endpoints already use.
- (Minimum) Just `limit: int = Query(100, le=500)` and pipe to repo.

### Frontend impact when fixed

- No FE rework needed beyond the existing call.

## REQ-2: POST /users/{id}/reset-password doesn't enforce X-Acting-Tenant-Id for SPA

### Severity
MEDIUM — security/audit gap. Functionally everything works, but
audit attribution is wrong when SPA forgets the header.

### Observed
Per CHANGES_APPLIED.md §2:
> **Super admin** — can reset anyone. Must pass `X-Acting-Tenant-Id`
> (since they have no home tenant).

Reality: the backend accepts the request without the header and returns
204. The audit row is then written with `tenant_id` from the SPA's
JWT (which is null), losing the cross-tenant attribution.

### Reproduction
```bash
ACCESS=$(login as admin@raniacone.com)
TARGET_UID=<some user id in any tenant>

curl -X POST http://localhost:8000/api/v1/users/$TARGET_UID/reset-password \
  -H "Authorization: Bearer $ACCESS" \
  -H "Content-Type: application/json" \
  -d '{"new_password":"NewPass1234!"}'
# Returns 204. Should return 400 TENANT_REQUIRED per the doc.
```

### Root cause analysis (located in source)

`packages/auth/app/api/v1/users.py`, lines 161–186:

```python
@router.post("/{user_id}/reset-password", status_code=204, response_class=Response,
             dependencies=[Depends(require_perm("auth.users.write"))])
async def admin_reset_password(
    user_id: UUID,
    body: AdminResetPasswordRequest,
    db: AsyncConnection = Depends(get_db),
    claims: dict = Depends(get_current_user_required),
    tid: Optional[UUID] = Depends(get_optional_acting_tenant_id),  # ← OPTIONAL
) -> Response:
    target = await user_repo.get_by_id(db, user_id)
    if not target:
        raise NotFoundError("User not found", code="USER_NOT_FOUND")
    _ensure_same_tenant(target, tid, bool(claims.get("sa")))
    # ↑ This early-returns for super admin. tid can be None and the SPA
    # can reset anyone's password without ever specifying which tenant
    # they're acting as. Audit row gets the SPA's null tenant_id.
    await user_service.admin_reset_password(...)
    return Response(status_code=204)
```

Two problems:
1. `get_optional_acting_tenant_id` lets SPA proceed with `tid=None`.
2. No `audit_super_admin_override` call, so cross-tenant overrides
   are NOT logged here, unlike every other §4 write endpoint.

### Proposed patch (drop-in replacement)

`packages/auth/app/api/v1/users.py`:

```python
@router.post("/{user_id}/reset-password", status_code=204, response_class=Response,
             dependencies=[Depends(require_perm("auth.users.write"))])
async def admin_reset_password(
    user_id: UUID,
    body: AdminResetPasswordRequest,
    db: AsyncConnection = Depends(get_db),
    claims: dict = Depends(get_current_user_required),
    tid: UUID = Depends(get_acting_tenant_id),       # ← strict (was: get_optional)
) -> Response:
    target = await user_repo.get_by_id(db, user_id)
    if not target:
        raise NotFoundError("User not found", code="USER_NOT_FOUND")
    _ensure_same_tenant(target, tid, bool(claims.get("sa")))
    # NEW: audit cross-tenant overrides like every other §4 write endpoint.
    await audit_super_admin_override(
        db,
        target_tenant_id=tid,
        endpoint="POST /users/{id}/reset-password",
    )
    await user_service.admin_reset_password(
        db,
        target_user_id=user_id,
        new_password=body.new_password,
        actor_id=UUID(claims["sub"]),
    )
    return Response(status_code=204)
```

### Why this is safe

- **Tenant admins**: unchanged — `get_acting_tenant_id` returns their JWT's
  `tid`. No 400.
- **SPA WITH header**: same as today — reset proceeds. New audit row.
- **SPA WITHOUT header**: now 400 `TENANT_REQUIRED` (was: silently 204).
  Matches §2 spec.
- **Existing 7 tests in `test_admin_reset_password.py`**: all assume
  the SPA sends the header (per the doc), so they keep passing.

### Test to add

```python
async def test_super_admin_reset_password_without_header_returns_400(
    client, sa_access_token, target_user_id,
):
    resp = await client.post(
        f"/users/{target_user_id}/reset-password",
        headers={"Authorization": f"Bearer {sa_access_token}"},
        json={"new_password": "Anything1!"},
    )
    assert resp.status_code == 400
    assert resp.json()["code"] == "TENANT_REQUIRED"
```

### Frontend impact

None today — the FE always passes `X-Acting-Tenant-Id` from
super-admin contexts (`/platform/users` and `/platform/tenants/[id]`
both use `userService.adminResetPassword(uid, pwd, actingTenantId)`).
Once the backend tightens, no FE rework is needed.

#### 2. New repo helpers in `packages/auth/app/repositories/rbac_repo.py`

```python
async def update_role(
    conn: AsyncConnection,
    role_id: UUID,
    tenant_id: UUID,
    *,
    name: Optional[str] = None,
    description: Optional[str] = None,
) -> Optional[dict[str, Any]]:
    """Tenant-scoped role update. Returns updated row, or None if no
    matching role in the given tenant."""
    sets, args = [], []
    if name is not None:
        sets.append("name = %s"); args.append(name)
    if description is not None:
        sets.append("description = %s"); args.append(description)
    if not sets:
        return await get_role_by_id(conn, role_id, tenant_id)
    sets.append("updated_at = now()")
    args.extend([role_id, tenant_id])
    async with conn.cursor(row_factory=dict_row) as cur:
        await cur.execute(
            f"UPDATE roles SET {', '.join(sets)} "
            f"WHERE id = %s AND tenant_id = %s "
            f"RETURNING {ROLE_COLS}",
            tuple(args),
        )
        return await cur.fetchone()


async def delete_role(
    conn: AsyncConnection,
    role_id: UUID,
    tenant_id: UUID,
) -> bool:
    """Tenant-scoped role delete. Returns True if a row was deleted."""
    async with conn.cursor() as cur:
        await cur.execute(
            "DELETE FROM roles WHERE id = %s AND tenant_id = %s",
            (role_id, tenant_id),
        )
        return cur.rowcount > 0


async def count_users_with_role(
    conn: AsyncConnection,
    role_id: UUID,
) -> int:
    """How many user_roles rows reference this role. Used to block
    deletion of roles still in use."""
    async with conn.cursor() as cur:
        await cur.execute(
            "SELECT COUNT(*) FROM user_roles WHERE role_id = %s",
            (role_id,),
        )
        row = await cur.fetchone()
        return int(row[0]) if row else 0
```

#### 3. New service methods in `packages/auth/app/services/rbac_service.py`

```python
async def update_role(
    conn: AsyncConnection,
    *,
    role_id: UUID,
    tenant_id: UUID,
    name: Optional[str],
    description: Optional[str],
    actor_id: Optional[UUID] = None,
) -> dict[str, Any]:
    async with conn.transaction():
        before = await rbac_repo.get_role_by_id(conn, role_id, tenant_id)
        if not before:
            raise NotFoundError("Role not found", code="ROLE_NOT_FOUND")
        if before.get("is_system"):
            raise ConflictError(
                "System roles cannot be modified",
                code="SYSTEM_ROLE_IMMUTABLE",
            )
        row = await rbac_repo.update_role(
            conn, role_id, tenant_id,
            name=name, description=description,
        )
        await audit_repo.log_audit(
            conn, tenant_id=tenant_id, table_name="roles",
            record_id=role_id, action="UPDATE",
            old_data=row_to_json(before),
            new_data=row_to_json(row),
            changed_by=actor_id,
            request_id=get_request_id(),
        )
        return row


async def delete_role(
    conn: AsyncConnection,
    *,
    role_id: UUID,
    tenant_id: UUID,
    actor_id: Optional[UUID] = None,
) -> None:
    async with conn.transaction():
        before = await rbac_repo.get_role_by_id(conn, role_id, tenant_id)
        if not before:
            raise NotFoundError("Role not found", code="ROLE_NOT_FOUND")
        if before.get("is_system"):
            raise ConflictError(
                "System roles cannot be deleted",
                code="SYSTEM_ROLE_IMMUTABLE",
            )
        # Block delete if anyone is currently assigned this role.
        # Otherwise we orphan user_roles rows and confuse RBAC.
        in_use = await rbac_repo.count_users_with_role(conn, role_id)
        if in_use > 0:
            raise ConflictError(
                f"Role is assigned to {in_use} user(s) — unassign first",
                code="ROLE_IN_USE",
                field_errors={"role_id": "in use"},
            )
        await rbac_repo.delete_role(conn, role_id, tenant_id)
        await audit_repo.log_audit(
            conn, tenant_id=tenant_id, table_name="roles",
            record_id=role_id, action="DELETE",
            old_data=row_to_json(before),
            changed_by=actor_id,
            request_id=get_request_id(),
        )
```

### Where the spec lives
`frontend/Resources/INVOICE_BACKEND_SPEC.md` — comprehensive,
covers:

- §1 Overview (state machine, purpose)
- §2 GST logic explained with worked examples (intra-state CGST+SGST
  vs inter-state IGST, discounts, cess, rounding rule)

- §7 Number Series wiring

- §9 Out of scope (e-Invoice IRN / e-Way Bill — Phase 4)
- §10 What FE has built (so backend dev knows nothing is FE-blocked)
- §11 Suggested ship sequence

### GST math — must match the FE port

The frontend already has a complete GST math implementation at
`src/lib/gst.ts` with 22 passing unit tests. The backend
implementation MUST agree to the rupee on every example in §2.4
of the spec. Same rounding rule (per-line, 2dp, half-up).

Specific cases the backend tests must cover (full list in §8):

- intra-state → CGST + SGST (50/50 split of total rate)
- inter-state → IGST (full rate, no CGST/SGST)
- discount applied BEFORE tax
- cess added on top of GST
- mixed-rate invoice (different lines at different rates)
- standard rates: 0, 0.25, 3, 5, 12, 18, 28
- non-standard rates accepted but warned

### Side effects on Post (transactional)

For each line:
1. Check qty_available >= quantity at the relevant location
2. If insufficient: `422 INSUFFICIENT_STOCK` with field_errors
3. Create OUT movement referencing the invoice
4. Update balance qty_on_hand and qty_available

On Cancel: reversal movements (paired IN with reference_movement_id),
plus reversal of ledger entries (Phase 3 — out of scope here).

### Permissions (already seeded in FE demo fixtures)

```

inventory.invoices.write   — create / edit drafts
inventory.invoices.post    — lock a draft, create movements
inventory.invoices.cancel  — reverse a posted invoice
```

Each is a separate gate so tenants can build any role mix
(e.g. "Sales Clerk: read+write" / "Billing Lead: all four"). The
existing `/admin/roles/[id]` permission grid handles it without
schema changes.

### Frontend impact when backend lands

Zero. The FE service layer (`src/services/invoices.service.ts`)
uses the production api-client; demo mode just swaps the axios
adapter for an in-memory mock. When backend ships:

1. Set `NEXT_PUBLIC_DEMO_MODE=false` in `.env.local`
2. FE hits real endpoints

No FE redeploy, no schema migration on the FE side, no flag flips

### Promote-to-invoice — the cross-module action

`POST /challans/{id}/promote-to-invoice` is the headline. Body
takes `place_of_supply` (drives GST split), optional `due_date`,
`invoice_number`, `remarks`. Server:

1. Validates challan is posted + unbilled.
2. Creates a draft invoice with `challan_id` back-link.
3. Copies every challan line into an invoice line, computing GST
   per INVOICE_BACKEND_SPEC.md §2 using `place_of_supply` vs tenant
   `state_code`.
4. Falls back to `items.hsn_code` / `items.default_tax_rate_pct`
   for any line missing those values.
5. Marks the challan `is_billed=true`, `invoice_id=<new>`.
6. Returns the new invoice.

### Critical invariant — no double stock OUT

Stock OUT is created on **challan post** (movements with
`reference_doc_type='challan'`). When that challan's invoice is
later **posted**, the invoice post handler MUST detect
`invoice.challan_id IS NOT NULL` and **skip** the stock OUT step.
The goods already left the warehouse on the challan — the invoice
is only the financial entry at that point.

This is a small change to the existing REQ-4 invoice post handler:

```python
# In invoice_service.post():
if invoice.challan_id is None:
    create_out_movements_and_decrement_balance(invoice)
# else: stock already moved on the challan post — skip
```

Cancellation symmetry:
- **Challan cancel** is forbidden if `is_billed=true` (must cancel
  the invoice first). Returns `409 CHALLAN_BILLED`.
- **Invoice cancel** must NOT create reversal movements when
  `invoice.challan_id IS NOT NULL` (the movements belong to the
  challan, not the invoice). Movement reversal happens on challan
  cancel.

### The headline mechanic — Employee Float

When a customer GPays a salesman personally instead of the company
UPI, the receipt records:

```
Dr. employee_float[ramesh]      8500.00     ← Ramesh holds it
Cr. party_receivable[mehta]     8500.00     ← Mehta's invoice settles
```

That float gets net-settled at salary time:

```
Salary due:                       50000.00
Less: float held                – 8500.00
Net payout:                       41500.00
```

Storage is double-entry; UX collapses it on the salary screen.
Constraint: receipts where mode=gpay AND payee_employee_id is set
must use that employee's float account as account_id (server-side
enforced as `EMPLOYEE_FLOAT_REQUIRED` 422).

### Architectural call — confirm before building

**Double-entry under the hood, multi-ledger UX on top.** Every
business event writes paired debit/credit rows to ledger_entries.
The UI presents these as separate "ledgers" by account type. This
saves a rewrite at Phase 3.5 (expenses, salary, P&L all need
double-entry).

If the backend dev disagrees and wants a flat-transactions model,
flag NOW before migrations land — switching later costs.

### Critical invariants (DO NOT skip in tests)

1. For every `group_id`, SUM(debit) = SUM(credit). Append-only,
   never UPDATE/DELETE ledger_entries — reversals are new rows
   with sign-flipped amounts and the same group_id.
2. `payments.allocated_amount = SUM(payment_allocations WHERE
   payment_id=...)`. Maintain in same transaction.
3. `financial_accounts.current_balance` is denormalised — must be
   updated in the same transaction as every ledger_entries
   insert. Drift here breaks the /money/debtors page.
4. Employee gpay constraint: receipt with mode=gpay AND
   payee_employee_id set MUST have account_id = that employee's
   float_account_id. 422 if not.

### The big architectural calls

**1. Vendor bills are FINANCIAL ONLY — no stock movement.**
A bill_post writes the AP ledger entry but does NOT create stock
IN movements. Stock IN happens through the GRN/PO flow (existing
documents module). This separates physical inventory tracking
(GRN) from financial accounting (bill).

**2. Expense categories own their own ledger account.**
Each expense_category row has a 1:1 `expense_account_id` pointing
at a financial_accounts row of type expense_category. Server
auto-creates the account on category insert. Pays off at report
time — "How much did we spend on Petrol this quarter?" is one
query against ledger_entries WHERE account_id = X.

**3. Salary closes the float loop.**
Net = gross − float_held. On post:
```
Dr. salary_expense                   gross_salary
Cr. employee_float[employee]         float_held    ← clears float
Cr. paid_from_account_id             net_paid       ← cash leaves
```
Server re-validates float at post time and rejects with
SALARY_FLOAT_DRIFT if the balance moved between draft and post.
Constraint: one posted salary per (employee, period_month) —
prevents accidental double-pay.

### Cross-module wiring (the tricky part)

1. **Bill post** writes a 3-row ledger entry: Dr expense (per-line
   override or default purchase_expense) + Dr gst_input + Cr
   party_payable[vendor]. Reversal on cancel.

2. **Expense post** writes a 2-row entry: Dr expense_category +
   Cr paid_from. Reversal on cancel.

3. **Salary post** writes a 2 or 3-row entry: Dr salary_expense
   + (optional) Cr employee_float + Cr paid_from. Skip the float
   row when float_held=0.

4. **Payment allocation** to a bill increments
   `vendor_bills.allocated_amount` (cached counter) but does NOT
   write new ledger entries — the original payment's posting
   already settled the party_payable. Same pattern as receipt-
   to-invoice allocation.

5. **Cancel-with-payments rule**: bills with allocated_amount > 0
   cannot be cancelled. Caller must cancel/unallocate the
   payments first. Returns `409 BILL_HAS_PAYMENTS`.

**Critical invariants documented in spec:**

   lives backend. No FE/DB drift possible.
3. **Commit re-validates** — last line of defence against time-gap
   races (preview blessed → DB state changed → commit). Re-validation
   failures land in `errors[]`, no rollback, partial success allowed.
4. **Templates are drift-locked** — header row in template MUST match
   keys validator + commit expect. One source of truth.
5. **Within-file uniqueness** — duplicates in the same upload flagged
   on BOTH rows during preview.
6. **Numbers/booleans are strings** — CSV is text; coerce explicitly.
   Reject obvious fat-finger errors (`"Y"`/`"₹100"`) at validator.
7. **Period-close interaction** — `opening_balances` rows with
   `as_of_date <= period_close.lock_before_date` rejected with 423
   per Phase 5 middleware.
8. **Stock balance idempotent** — re-upload overwrites existing
   `(item_id, location_id)` balance instead of duplicating.

- `src/services/documents.service.ts` — createLine signature accepts
  thickness_mm/size_code.
- `src/components/layout/sidebar.tsx` — "Item Pricing" entry under
  Master Data (admin-gated).

**Backend endpoints requested (full spec at
`Resources/PHASE13_BACKEND_SPEC.md`):**
```

GET  /pricing-rules?item_id=&active_only=  — list rules
GET  /pricing-rules/lookup                 — auto-fill helper for line forms
POST /pricing-rules                        — create new (auto-closes prior)
```

**Critical invariants:**

1. **Single active rule** per (tenant, item, thickness, size) at any
   moment. Enforced by:
   ```
   CREATE UNIQUE INDEX idx_pricing_rules_one_active
     ON item_pricing_rules (tenant_id, item_id, thickness_mm, size_code)
     WHERE valid_until IS NULL;
   ```

2. **POST is transactional** — new rule's insert + prior rule's
   `valid_until = (new.valid_from − 1 day)` happen in one transaction.

3. **Lines snapshot the price** — historical lines never change when
   a new pricing rule is created. The line's `unit_price` is the
   record of truth for that document.

4. **Auto-lookup contract** — on line POST, if `thickness_mm` and
   `size_code` are supplied AND `unit_price` is missing/0, server
   runs the lookup and uses the rule's price.

5. **Per-tenant dimensions** — `item_dimensions` is tenant-scoped.
   Nova Bond has 12 (4 × 3); other tenants seed their own.

**Schema changes:**

```sql
CREATE TABLE item_dimensions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  thickness_mm INTEGER NOT NULL,
  size_code TEXT NOT NULL,
  label TEXT NOT NULL,
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, thickness_mm, size_code)
);

CREATE TABLE item_pricing_rules (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  item_id UUID NOT NULL REFERENCES items(id),
  thickness_mm INTEGER NOT NULL,
  size_code TEXT NOT NULL,
  sale_price NUMERIC(14,2) NOT NULL CHECK (sale_price >= 0),
  valid_from DATE NOT NULL,
  valid_until DATE,
  notes TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by UUID REFERENCES users(id),
  CHECK (valid_until IS NULL OR valid_until >= valid_from)
);

CREATE INDEX idx_pricing_rules_lookup
  ON item_pricing_rules (tenant_id, item_id, thickness_mm, size_code, valid_from DESC);

CREATE UNIQUE INDEX idx_pricing_rules_one_active
  ON item_pricing_rules (tenant_id, item_id, thickness_mm, size_code)
  WHERE valid_until IS NULL;

-- Line table extensions (4 ALTERs)
ALTER TABLE invoice_lines     ADD COLUMN thickness_mm INTEGER, ADD COLUMN size_code TEXT;
ALTER TABLE challan_lines     ADD COLUMN thickness_mm INTEGER, ADD COLUMN size_code TEXT;
ALTER TABLE document_lines    ADD COLUMN thickness_mm INTEGER, ADD COLUMN size_code TEXT;
ALTER TABLE vendor_bill_lines ADD COLUMN thickness_mm INTEGER, ADD COLUMN size_code TEXT;
```

**Permissions:** reuses `inventory.master_data.read` for view (already
exists) and `inventory.master_data.write` for create (add this code
to permissions if not yet seeded).

**Suggested backend ship order:**
1. Schema migrations (low risk, schema-only)
